# Deribit Price Tracker

Сервис для автоматического сбора и хранения цен криптовалют BTC/USD и ETH/USD с биржи Deribit. 
Данные собираются каждую минуту и сохраняются в PostgreSQL. Доступ к историческим данным предоставляется через REST API.

# Технологии

    FastAPI - современный веб-фреймворк для создания API

    PostgreSQL 15 - реляционная база данных для хранения цен

    Redis 7 - брокер сообщений для Celery

    Celery - асинхронная очередь задач для периодического сбора данных

    SQLAlchemy 2.0 - ORM для работы с базой данных (паттерн Repository)

    Aiohttp - асинхронный HTTP клиент для запросов к Deribit API

    Docker & Docker Compose - контейнеризация и оркестрация сервисов

    Pydantic - валидация данных и управление настройками

# Архитектура

Проект состоит из трех Docker-контейнеров:
    1.db (PostgreSQL) - хранение исторических данных цен

    2.redis (Redis) - брокер сообщений для Celery

    3.app (FastAPI + Celery) - основное приложение:

        FastAPI приложение с REST эндпоинтами

        Celery worker для выполнения задач

        Celery beat для планирования периодических задач

# Установка и запуск

Предварительные требования:

    1. Установленные Docker и Docker Compose (версия 3.8+)

    2. Git

    3. Свободные порты: 5432, 6379, 8000

1.  git clone <https://github.com/pamparam130/deribit-price-tracker>
    cd deribit-price-tracker

2. Создать файл .env:
DATABASE_URL=postgresql://user:password@db:5432/deribit_prices
REDIS_URL=redis://redis:6379/0
API_V1_PREFIX=/api/v1
PROJECT_NAME=Deribit Price Tracker
DERIBIT_API_URL=https://www.deribit.com/api/v2

3.  docker-compose up -d --build
    docker-compose ps

# Design Desicions

1. Архитектурные паттерны
Паттерн Repository

Использован для абстракции доступа к данным. Преимущества:

    Централизация - вся логика работы с БД в одном месте

    Тестирование - легко подменять на mock-объекты

    Гибкость - возможность замены ORM или БД без изменения бизнес-логики

    Чистота кода - разделение ответственности (Separation of Concerns)

class PriceRepository:
    def get_by_ticker(self, ticker): ...
    def get_latest_by_ticker(self, ticker): ...
    def get_by_ticker_and_date_range(self, ticker, start, end): ...

Dependency Injection

Использование зависимостей FastAPI для внедрения репозитория:

    Автоматическое управление жизненным циклом

    Легкое тестирование с переопределением зависимостей

    Четкие контракты между компонентами

Separation of Concerns (Разделение ответственности)

    API Layer (endpoints) - только обработка HTTP запросов

    Repository Layer - только работа с данными

    Service Layer - бизнес-логика (клиент Deribit)

    Task Layer - асинхронные задачи Celery

    Schema Layer - валидация и сериализация данных

2. Работа с базой данных
Индексы и производительность

-- Оптимизация частых запросов
CREATE INDEX ix_prices_ticker ON prices (ticker);      -- для фильтрации по валюте
CREATE INDEX ix_prices_timestamp ON prices (timestamp); -- для сортировки и диапазонов
CREATE INDEX ix_prices_ticker_timestamp ON prices (ticker, timestamp); -- для составных запросов

Выбор типов данных

    DECIMAL/NUMERIC для финансовых данных (точность важнее скорости)

    TIMESTAMP WITH TIME ZONE для хранения времени в UTC

    SERIAL для автоинкрементных первичных ключей

Connection Pooling

    Настройка пула соединений для оптимальной производительности

    Предотвращение исчерпания соединений к БД

    Автоматическое восстановление оборванных соединений

3. Асинхронность и конкурентность
Aiohttp для HTTP клиента

    Неблокирующие запросы к Deribit API

    Параллельный сбор цен для нескольких валют

    Экономия системных ресурсов (один поток на множество запросов)

async def get_prices(self):
    tasks = [
        self.get_index_price('btc'),
        self.get_index_price('eth')
    ]
    return await asyncio.gather(*tasks)  # параллельное выполнение

Celery архитектура

    Beat - планировщик, отправляет задачи в Redis

    Worker - исполнители, забирают задачи из Redis

    Масштабирование - можно запускать несколько воркеров

    Отказоустойчивость - задачи не теряются при падении воркера

4. Конфигурация и безопасность

    Код в репозитории - вся конфигурация в коде

    Зависимости - явно указаны в requirements.txt

    Конфигурация - через переменные окружения

    Бэкенды - PostgreSQL и Redis как подключаемые сервисы

    Сборка и запуск - разделены (Docker слои)

Безопасность

    Секреты только в .env (исключен из git)

    Изоляция контейнеров через Docker сети

    Непривилегированные пользователи в контейнерах

    Валидация всех входных данных

5. Масштабирование и производительность
Горизонтальное масштабирование

    API - можно запустить несколько экземпляров за балансировщиком

    Celery workers - можно увеличить количество для параллельной обработки

    Redis - кластеризация для больших нагрузок

    PostgreSQL - репликация чтения для аналитики

Кэширование

    Redis как основной кэш (но в данном проекте используется как брокер)

    Потенциально можно кэшировать частые запросы к API

    Кэширование последних цен для уменьшения нагрузки на БД


6. Выбор технологий
FastAPI вместо Django/Flask

    Асинхронность из коробки

    Автоматическая документация OpenAPI

    Валидация через Pydantic

    Высокая производительность (на уровне Node.js)

Celery вместо простого threading

    Распределенная архитектура

    Персистентность задач

    Мониторинг и управление

    Гарантия выполнения

SQLAlchemy вместо сырых запросов

    Абстракция от конкретной БД

    Безопасность (защита от SQL injection)

    Удобство миграций

    Работа с объектами вместо кортежей

Docker вместо виртуального окружения

    Изоляция окружения

    Воспроизводимость на любой машине

    Простота развертывания

    Масштабирование
